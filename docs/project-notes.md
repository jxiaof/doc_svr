# 项目维护记录

更新时间：2026-05-13

## 已确认的现状

- 当前项目的核心职责是聚合 public 下的静态页面，不再是 Markdown 博客渲染站点
- 首页模板、共享样式和 public 页面都通过 embed.FS 打包进单个二进制
- 构建默认依赖 vendor 目录，适合内网或离线环境

## 当前不太合理的地方

### 1. 项目说明长期滞后于代码

此前 README 仍在描述 content/posts、Goldmark 和博客场景，但仓库已经转成静态页面入口服务。这类漂移会直接影响接手成本。

处理状态：已在 README 中对齐当前结构。

### 2. 运行口径存在不一致

之前文档和脚本默认 3000 端口，但主程序默认值已经切到 4000，容易造成“服务已启动但访问错端口”的误判。

处理状态：已统一为 4000，并补充了 `--port` 参数与首页静态生成命令。

### 3. 静态页面全部 embed 进二进制

这让部署很方便，但 public 里的大型控制台页面会直接推高二进制体积和构建时间。后续如果页面越来越多，需要评估是否把部分资源改成外置挂载。

处理建议：持续观察二进制大小；超过交付阈值后，再评估拆分资源策略。

### 4. 自动化验证偏少

当前主要依赖 go build ./... 做校验，没有针对路由发现、首页渲染、健康检查的自动化测试。

处理建议：优先补 internal/site 的单元测试和最小 HTTP smoke test。

### 5. scripts 目录职责不够清晰

scripts/run.sh 和 build.sh 有重叠；scripts/install.sh 是强环境绑定的 Ubuntu 安装脚本，不属于项目核心运行链。

处理建议：后续区分 dev helper 与真正的项目脚本，必要时把环境安装脚本移到 docs/ops。

### 6. deploy 目录当前为空

目录已经保留，但没有任何可执行部署文件，容易让人误以为有现成部署方案。

处理建议：短期内二选一。

- 要么补上容器运行示例或清单
- 要么删掉这个空目录，避免误导

### 7. Docker 构建依赖外部镜像仓库

当前本地验证里，make docker-build 被 Docker Hub 拉取超时阻断。这不是 dockerfile 语法问题，但会影响 CI 和新环境初始化。

处理建议：

- 在团队环境准备镜像代理或私有缓存
- CI 中预热基础镜像

## 建议的后续优先级

### P1

- 为 internal/site 增加基本测试
- 统一共享导航样式，减少服务端 HTML 字符串注入复杂度
- 明确 deploy 目录去留

### P2

- 梳理 scripts 目录职责
- 评估 public 资源 embed 体积增长趋势
- 补充容器运行与发布约定

### P3

- 如果静态页数量继续增长，引入更明确的元数据配置，而不是只靠目录名推断标题
